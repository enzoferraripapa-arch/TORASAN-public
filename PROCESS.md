## 1. 目的と適用範囲

本文書は **ISO 26262 をベース** とし、**Automotive SPICE プロセスフレームワーク** に従った
組み込み安全関連ソフトウェアの開発プロセスを定義する。

製品ごとの規格差異は `project.json` の設定により **パッケージを切り替えて** 対応する。
本プロセス自体の開発・管理にも同プロセスを適用する（自己参照適用）。

### 1.1 ベース規格

| 規格 | 役割 |
|------|------|
| **ISO 26262:2018** | 機能安全要求のベース（Part 3〜6を主に適用） |
| **Automotive SPICE v3.1** | プロセスフレームワーク・能力レベル定義 |
| **MISRA C:2012** | コーディング規約（全製品共通） |

### 1.2 製品オーバーライド規格（project.json で選択）

| 設定値 | 規格 | 適用製品例 | ISO 26262 相当 |
|--------|------|----------|--------------|
| `ISO26262` | ISO 26262 フル | 車載ECU | ASIL A〜D |
| `IEC60730` | IEC 60730-1 Class B | 家電制御 | ASIL B 相当 |
| `IEC61508` | IEC 61508 SIL2 | 産業機器 | ASIL C 相当 |
| `null` | ISO 26262 のみ | カスタム | project.jsonのasil準拠 |

### 1.3 設計原則

| No | 原則 | 検証方法 |
|----|------|---------|
| P-01 | 全ルールは自動検証手段を持つこと | ルール定義時に検証コマンドを併記 |
| P-02 | 数値の Single Source of Truth は `project.json` | grep照合スクリプト |
| P-03 | 型・関数名は静的解析で検証 | GCC -Wall -Werror / cppcheck |
| P-04 | 文書⇔コード間の関数名・数値は自動照合 | フェーズゲートで照合スクリプト実行 |
| P-05 | 各フェーズには完了ゲート条件があり、全PASSでないと次に進めない | ゲート判定スクリプト |
| P-06 | バックアップは変更前に必ず取得し、ロールバック可能にすること | archive/vX/ + restore.sh |
| P-07 | 本プロセス開発自体も本プロセスに従い記録を残すこと | process_records/ |

### 2.2 能力レベル定義と達成条件

| レベル | 名称 | 意味 | 達成条件 |
|--------|------|------|---------|
| 0 | Incomplete | プロセス未実施または部分的 | — |
| 1 | Performed | プロセスが実施され成果物が存在する | PA 1.1 ≧ L |
| 2 | Managed | 計画・監視・調整が行われている | PA 1.1 = F, PA 2.1 ≧ L, PA 2.2 ≧ L |
| 3 | Established | 組織標準プロセスが定義・展開されている | Level 2 + PA 3.1 ≧ L, PA 3.2 ≧ L |

**評価スケール**: N = 未達成(0-15%), P = 部分達成(16-50%), L = 概ね達成(51-85%), F = 完全達成(86-100%)

## 4. 規格選択ルール

### 4.1 project.json 設定に基づく規格適用

```
project.json
├── standard.base              = "ISO26262"  → プロセスフレームワーク（常時適用）
├── standard.asil              = "B"         → プロセス厳密度の決定
├── standard.product_override  = "IEC60730"  → 製品規格（実際の安全要求源泉）
└── standard.product_override_class = "B"    → 製品安全分類（Class A/B/C）
```

規格の組み合わせに応じた安全機構の選択は、各プロジェクトリポの `src/` 内で実装する。
TORASAN フレームワークはプロセス定義・スキル・ナレッジを提供し、実装コードは各PJリポに配置する。

### 4.2 SPF（Single Point of Failure）対策方針

安全関連 BSW モジュールは以下を必須とする：

| 要求ID | 要求内容 | 検証方法 |
|--------|---------|---------|
| SPF-01 | 全公開関数は戻り値で異常を通知すること（`Std_ReturnType`） | コードレビュー |
| SPF-02 | 初期化失敗時は安全状態に遷移すること | ユニットテスト |
| SPF-03 | タイムアウト監視を持つこと（WDT連携） | wdt_mgr統合確認 |
| SPF-04 | ハードウェア依存コードはHAL経由のみ | grep: 直接レジスタアクセス禁止 |

## 5. 型システムルール（全ソースコード共通）

### 5.1 必須ルール

| ルールID | 内容 | 検証方法 |
|---------|------|---------|
| T-01 | `bool`/`true`/`false`/`<stdbool.h>` 使用禁止 | `grep -rn 'bool\|stdbool\|true\|false' src/` |
| T-02 | 全整数型は固定幅型: `uint8_t`, `uint16_t`, `uint32_t`, `int8_t`, `int16_t`, `int32_t` | `grep -rn '\bint\b\|\blong\b\|\bshort\b\|\bchar\b' src/` |
| T-03a | `float`同士の`==` `!=`比較禁止（許容誤差比較のみ） | `grep -rn '==.*f\b\|!=.*f\b' src/` |
| T-03b | `double`使用禁止 | `grep -rn '\bdouble\b' src/` |
| T-03c | `<math.h>`使用禁止 | `grep -rn 'math\.h' src/` |
| T-04 | 論理値は `uint8_t` + `STD_TRUE(1U)` / `STD_FALSE(0U)` | std_types.h必須include確認 |
| T-05 | enum値は全て明示値付与（`VALUE = 0U`形式） | パターンgrep |
| T-06 | `union`使用禁止 | `grep -rn '\bunion\b' src/` |
| T-07 | 動的メモリ禁止: `malloc`, `calloc`, `realloc`, `free` | `grep -rn 'malloc\|calloc\|realloc\|\bfree\b' src/` |
| T-08 | 再帰呼び出し禁止 | cppcheck + コールグラフ |
| T-09 | `goto`使用禁止 | `grep -rn '\bgoto\b' src/` |
| T-10 | 暗黙の型変換禁止（全キャスト明示） | GCC -Wconversion -Wsign-conversion |
