## 2. PROCESS.md のレビュー

### 2.1 良い点（Strengths）

- **Vモデルの完全なフェーズ構成（PH-01〜PH-15）** が明確に定義されており、各フェーズのISO 26262参照条項も明記されている
- **要件ID体系（SG/FSR/TSR/SR/DR/SA/UT/TC）** が体系的で、双方向トレーサビリティの仕組みが整備されている
- **探索モード / 認証モード** の切り分けにより、プロトタイプ段階と製品化段階の作業レベルを明確に区別
- **前提変更の影響マトリクス** が実用的で、MCU変更やASIL変更時の影響範囲が一目でわかる
- **多規格対応マトリクス**（ISO 26262 / IEC 61508 / IEC 60730 / IEC 62061 / JIS B 9705 / EU機械規則）が充実
- **作業トレース** の出力フォーマットが具体的で、作業の透明性が高い
- **ツール適格性確認** でClaude CodeをTCL3と適切に分類し、方法A（全成果物レビュー）を採用

### 2.2 改善提案（Issues & Recommendations）

#### 🔴 重要度：高

**I-01：CLAUDE.md と PROCESS.md の役割の曖昧さ**
CLAUDE.md は「起動設定ファイル」と位置づけているが、PROCESS.md の「起動時の動作」セクション（STEP 1〜5）と CLAUDE.md の指示表が重複しており、どちらが正かが不明確になる恐れがある。
→ **推奨：** CLAUDE.md はルーティング（振り分け）のみに徹し、具体的な動作定義はすべて PROCESS.md / INGEST.md に集約する旨を明記する。

**I-02：認証モード移行の具体的チェックリストがない**
「認証モードに移行できる状態か確認してください」の指示は定義されているが、移行判定基準（ゲートクライテリア）が明示されていない。
→ **推奨：** 以下の移行条件を明文化する：
- 全フェーズの `tbdCount = 0`
- 全フェーズの `unlinkedReqs = 0`
- 全【AI補完・要確認】タグの解消
- 全フェーズのレビュー記録が存在
- ツール適格性記録が完了
- FMEA/FTA が ASIL に応じて完了

**I-03：DIA（従属故障分析）への言及がない**
ISO 26262 Part 9（ASIL分解と従属故障分析）は、特にASIL C/D で重要だが、フェーズ構成に含まれていない。
→ **推奨：** PH-05（TSC）またはPH-06（システム設計）の中にDFA（Dependent Failure Analysis）の項目を追加するか、独立フェーズとして追加を検討する。

**I-04：安全妥当性確認（Safety Validation）の位置づけが弱い**
PH-15（機能安全アセスメント）はあるが、ISO 26262 Part 4 cl.9 の「安全妥当性確認」は「テスト仕様書」とは異なる概念。検証（Verification）と妥当性確認（Validation）の区別が曖昧。
→ **推奨：** PH-13/14 のテスト仕様書セクションに「これは検証であり、妥当性確認は人間が実機で実施する」旨を明記。PH-15 に Validation 確認項目を追加。

#### 🟡 重要度：中

**I-05：HW-SW インタフェース仕様書（HSI）が独立成果物として定義されていない**
ISO 26262 Part 6 cl.6 では HSI は重要な成果物。現在は PH-06 の中に埋もれている。
→ **推奨：** `docs/06_HSI.md` を独立成果物として追加し、PH-06 の出力物として明記する。

**I-06：project.json の `standards` の `level` フィールドに型の不整合がある可能性**
`"level": "ASIL-B"` と `"level": null` が混在。null の場合の処理ルールが未定義。
→ **推奨：** level のデフォルト値と型（文字列 or null）の扱いを明文化する。

**I-07：フェーズ間の依存関係が明示されていない**
探索モードでは「順序自由」だが、認証モードでの厳密な依存関係（例：PH-03完了なしにPH-04開始不可）がフロー図として示されていない。
→ **推奨：** 認証モード用のフェーズ依存グラフ（DAG）を追加する。

**I-08：Part 8 cl.12（ソフトウェアコンポーネントの適格性確認）への言及がない**
外部ライブラリ管理は別ファイル化されているが、PROCESS.md 側で SEooC（Safety Element out of Context）の判断フローが不足。
→ **推奨：** PH-09 の前提条件に「外部ライブラリの SEooC 評価完了」を追加。

**I-09：バージョン管理ルールに Git 運用との連携が未定義**
`.git` フォルダが存在するが、ブランチ戦略・タグ付けルール・ベースライン管理との紐付けがない。
→ **推奨：** Git タグと文書バージョン・ベースライン設定の対応ルールを追加する。

#### 🟢 重要度：低

**I-10：ダッシュボードの表示幅が環境依存**
ASCII罫線によるダッシュボードは、端末幅によっては崩れる可能性がある。
→ **推奨：** 将来的にはMarkdownテーブル形式への切り替えも検討。

**I-11：PH-14（車両統合テスト仕様書）が非車載プロジェクトで混乱を招く可能性**
IEC 61508 や IEC 60730 選択時に PH-14 がダッシュボードに表示されるのは違和感がある。
→ **推奨：** 車載以外の規格選択時は PH-14 を非表示またはスキップ扱いとするロジックを追加。

**I-12：コード例の言語がC限定**
コーディングガイドラインや安全タグの例がC言語のみ。C++やRust等への対応方針が未記載。
→ **推奨：** 対応言語のスコープを明記し、他言語のタグフォーマット例を追加。

## 3. INGEST.md のレビュー

### 3.1 良い点（Strengths）

- **段階的処理（スキャン→分類→再分類→ギャップ分析→再文書化）** の設計が実用的
- **流用価値の4段階評価**（⭐⭐⭐ / ⭐⭐ / ⭐ / ✗）が明快
- **機能・技術要素単位の再分類** は、プロジェクト横断での資産活用に有効
- **【AI補完・要確認】タグ** による補完箇所の可視化は安全面で重要
- **機密情報検出・重複ファイル検出** の特殊ケース対応が考慮されている
- **中断・再開機能**（ingest.json）が大量資産の処理に対応

### 3.2 改善提案（Issues & Recommendations）

#### 🔴 重要度：高

**J-01：ingest.json のスキーマ定義がない**
project.json はテンプレートが示されているが、ingest.json は言及のみで構造が未定義。
→ **推奨：** ingest.json のJSONスキーマテンプレートを追加する。

**J-02：再文書化時の「元資産との差分」管理が不十分**
F-002 → docs/08_SW安全要求.md への変換時、元資産のどの部分を流用し、どの部分をAIが補完したかのトレースが reuse_log.md だけでは粗い。
→ **推奨：** 再文書化した成果物内にインラインで「元資産：F-002 Section 3.2 から流用」等の出典タグを付与するルールを追加。

#### 🟡 重要度：中

**J-03：Office文書（.docx/.xlsx）の読み取り可否がClaude Codeの能力に依存**
INGEST.md は Claude Code がこれらを読める前提で書かれているが、実際の読み取り精度やレイアウト崩れへの対応が未記載。
→ **推奨：** 「Claude Code が正確に読み取れない場合」のフォールバック手順（ユーザーにテキスト貼り付けを依頼、PDF化して再読み込み等）を追加。

**J-04：アーカイブ（.zip等）の自動展開ルールが未定義**
ファイル種別判定表に `.zip` / `.tar` / `.gz` が「要展開」と書かれているが、展開手順や展開先が未定義。
→ **推奨：** `legacy/` 配下に展開するか `ingest_work/` に展開するかのルールを明記。

**J-05：STEP 3（機能要素再分類）からSTEP 4（ギャップ分析）への自動遷移条件がない**
各STEPの完了条件・次STEP移行条件が明示されていない。
→ **推奨：** 各STEPに Exit Criteria を追加する。

#### 🟢 重要度：低

**J-06：FE-カテゴリが固定で拡張性が低い**
FE-MOT / FE-COM 等のカテゴリが10種に固定されている。プロジェクト固有の機能要素を追加するルールがない。
→ **推奨：** `FE-USR-XXX`（ユーザー定義カテゴリ）の追加ルールを明記。

4. 検証/妥当性確認の区別明確化（I-04）

これらを対応すれば、探索モードでの運用開始には十分な状態と判断します。
