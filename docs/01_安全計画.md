## 1. プロジェクト概要と適用範囲

### 1.1 プロジェクト概要

| 項目 | 内容 |

## 2. 適用規格・安全レベル目標

### 2.1 規格体制（二重規格方式）

| 区分 | 規格 | 役割 |
|------|------|------|
| ベースプロセス | ISO 26262:2018 | V-model開発プロセス・安全分析手法の基盤 |
| 製品規格 | IEC 60730-1:2023 Class B | 家庭用電気器具の安全要求（最終適合対象） |
| プロセス成熟度 | Automotive SPICE v3.1 | 開発プロセスの能力レベル評価 |
| コーディング規約 | MISRA C:2012 | Cコード品質・安全性確保 |

### 2.2 安全レベル目標

| 項目 | 目標値 |
|------|--------|
| 製品安全分類 | IEC 60730 Class B（安全自動制御機能付き） |
| プロセス基盤 | ISO 26262 手法を活用（ASIL判定は非適用） |
| ソフトウェア信頼度レベル (TCL) | TCL3（最高レベル） |
| SPICE 能力レベル目標 | Level 2 |
| MISRA C 準拠 | 必須ルール100%、推奨ルール逸脱文書化 |

### 2.3 IEC 60730 Annex H 自動診断要件

| 診断項目 | 要件 | 手法 |
|---------|------|------|
| CPU レジスタ | 異常検出 | レジスタテスト |
| RAM | 破損検出 | March C（256Bブロック × 22、全域2.2秒） |
| Flash/ROM | 改ざん検出 | CRC32 |
| ADC | 入力異常検出 | ADC プラウジビリティチェック（電流100Hz/電圧10Hz） |
| WDT | プログラムフロー監視 | ウォッチドッグ（100ms TO / 50ms キック） |
| 電圧監視 | 電源異常検出 | 4.5V〜5.5V 範囲監視 |
| 電流監視 | 過電流検出 | 定格6A / 最大8A |
| 過速度 | モータ暴走検出 | 定格1200rpm / 閾値1500rpm |

## 6. サイバーセキュリティ対応方針

### 6.1 適用規格・法規

| 区分 | 名称 | 適用条件 | 現プロジェクトへの影響 |
|------|------|---------|-------------------|
| **法規** | UN-R155 (WP.29 CSMS) | 車載ECU | 車載パッケージ使用時に必須 |
| **法規** | EU Cyber Resilience Act (CRA) | デジタル要素を持つ全製品 (EU市場) | 2027年施行後に必須 |
| **法規** | EU RED 3.3(d)(e)(f) | WiFi/BT搭載機器 (EU市場) | ネットワーク接続ありの場合に必須 |
| **規格** | ISO/SAE 21434 | 車載サイバーセキュリティ | UN-R155 適合の実装手段 |
| **規格** | ETSI EN 303 645 | コンシューマーIoT | CRA/RED 適合の推定手段 |
| **規格** | IEC 62443 | 産業用制御システム | 産業機器パッケージ使用時 |

### 6.2 Safety-Security 並走方針

安全（Safety）とセキュリティ（Security）は V-model の各フェーズで並走する:

| フェーズ | Safety 活動 | Security 活動 | 統合スキル |
|---------|-----------|-------------|----------|
| PH-03 | HARA（ハザード分析） | TARA（脅威分析） | `/fmea` (safety/security モード) |
| PH-04 | FSC（機能安全コンセプト） | SecC（セキュリティコンセプト） | `/safety-concept` |
| PH-05 | TSC（技術安全コンセプト） | SecTSC（技術セキュリティ要求） | `/safety-concept` |
| PH-08 | SRS（SW安全要求） | SecSRS（SWセキュリティ要求） | `/srs-generate` |
| PH-10 | MISRA C 準拠 | CERT C / CWE 対策 | `/static-analysis` (flawfinder) |
| PH-11 | 安全テスト | セキュリティテスト | `/test-design` |
| PH-14 | Safety Case (GSN) | Security Case | `/safety-verify` |

### 6.3 セキュリティトレーサビリティ ID体系

| 接頭辞 | 名称 | 対応する Safety ID |
|--------|------|------------------|
| SecG-xx | セキュリティ目標 | SG-xx |
| SecR-xx | セキュリティ要求 | FSR-xx |
| SecTR-xx | 技術セキュリティ要求 | TSR-xx |
| SecSR-xx | SWセキュリティ要求 | SR-xx |
| SecTC-xx | セキュリティテストケース | TC-xx |
| TH-xx | 脅威 (Threat) | HZ-xx |

### 6.4 段階的導入計画

本プロジェクト（EXPLORE モード）では、セキュリティ対応を各フェーズで段階的に導入し、PH-15 で全体を見直す:

- **Phase 1 (PH-01〜PH-03)**: 規格体系の整理、TARA の枠組み定義
- **Phase 2 (PH-04〜PH-08)**: セキュリティ要求のトレーサビリティ確立
- **Phase 3 (PH-09〜PH-11)**: セキュアコーディング、脆弱性テスト
- **Phase 4 (PH-12〜PH-15)**: セキュリティケース統合、全体見直し

## 7. 確認措置・レビュー計画

### 7.1 フェーズゲート検証

各フェーズ完了時に PROCESS.md §3 のゲート条件を全て PASS することを必須とする。FAIL の場合は差し戻して是正する。

### 7.2 レビュー方針

| モード | レビュー方法 | 適用時期 |
|--------|-----------|---------|
| EXPLORE | TORA による自己レビュー + AI支援チェック | 現在 |
| CERTIFY | 独立検証 + 外部専門家レビュー | 将来移行時 |

### 7.3 レビュー重点項目

- IEC 60730 Annex H 適合性
- MISRA C:2012 準拠性
- 自動診断機能の仕様妥当性
- トレーサビリティの双方向完全性（Safety + Security）
- product_spec 数値の整合性
- セキュリティ要求の充足性（該当フェーズ）
- CWE/CERT C 脆弱性対策（コード生成フェーズ）

## 8. 変更管理方針

### 8.1 変更プロセス

1. **変更要求記録** → `project.json` changeLog に CHG-xxx として記録
2. **影響分析** → affectedPhases で影響範囲を特定
3. **承認** → TORA による承認
4. **実装・検証** → 該当フェーズの成果物を更新、ゲート再検証
5. **ドキュメント更新** → 関連ドキュメント版数更新

### 8.2 バージョン管理

| 対象 | 管理方法 |
|------|---------|
| ドキュメント | 文書番号 + 版数（本文書ヘッダに記載） |
| ソースコード | Git タグ + コミットハッシュ |
| プロジェクト状態 | project.json（Single Source of Truth） |
| 変更履歴 | project.json changeLog |

## 9. 進捗監視・逸脱対処

### 9.1 進捗監視方法

- `project.json` phases による各フェーズ status 追跡
- gateLog によるゲート検証結果の記録
- ダッシュボード（`/dashboard`）による可視化

### 9.2 逸脱対処手順

1. ゲート FAIL 検出 → problemLog に記録
2. 原因分析 → `/problem-resolve` で根本原因を特定
3. 是正計画策定 → 影響範囲と対処方法を定義
4. 是正実施 → 成果物を修正し、ゲート再検証
5. 確認 → TORA がクローズを承認
