# WSL Ubuntu への Claude Code 追加導入 — 多角検討議事録

**日時**: 2026-03-06
**議題**: WSL Ubuntu 24.04 に Linux 版 Claude Code を導入することで自由度・効率が向上するか
**参加エージェント**:
- **エージェントA**: 技術検証担当（実現可能性）
- **エージェントB**: 効率・メリット分析担当
- **エージェントC**: リスク・デメリット分析担当

---

## 1. 各エージェントの結論サマリ

| エージェント | 結論 | 一言要約 |
|------------|------|---------|
| **A（技術検証）** | 条件付き可能 | インストールは容易。サンドボックスが最大の技術的メリット。ただし /mnt/c は2-5倍遅い |
| **B（メリット）** | 導入推奨 | 真の並列実行、規格適合ツール(gcov/valgrind)、Docker対応で効果「高」 |
| **C（リスク）** | **やらない方がいい** | 致命的リスク4件（設定分裂/パス混在/Git競合/複雑性）。必要性の根拠なし |

## 2. 主要論点の対立構図

### 論点1: 並列実行の価値

| | エージェントB（推進派） | エージェントC（反対派） |
|--|----------------------|----------------------|
| 主張 | 独立インスタンスによる真の並列で40-50%時間短縮 | ボトルネックはユーザーの注意力。2ターミナル監視は非現実的 |
| 根拠 | サブエージェント(Task tool)は親プロセス内の逐次処理 | 対話型エージェントには人間の指示が必要 |
| **判定** | △ 理論上は正しいが… | ○ **実運用上はCの指摘が妥当** |

**議長メモ**: Claude Code の Task tool（サブエージェント）は既に並列実行を実現しており、2つ目のインスタンスを起動するまでもない場面がほとんど。「完全独立した長時間タスク」が必要なケースは限定的。

---

### 論点2: Linux ネイティブツールの価値

| | エージェントB（推進派） | エージェントC（反対派） |
|--|----------------------|----------------------|
| 主張 | valgrind/gcov/lcovが規格適合に直結。「便利」ではなく「必須」 | `wsl <command>` で Git Bash から WSL コマンド実行可能。Claude Code を WSL に常駐させる必要なし |
| 根拠 | ISO 26262 Part 6 がカバレッジ計測を要求 | 現状で cppcheck/clang-tidy/flawfinder は動作中 |
| **判定** | ○ ツール自体の価値は正当 | ○ **アクセス方法としてはCが合理的** |

**議長メモ**: Bが挙げたツール（valgrind, gcov, lcov）の価値は本物。しかしCの指摘通り、それらを使うために **Claude Code ごと WSL に入れる必要はない**。`wsl valgrind ./test` のように既存の Claude Code から呼び出せばよい。

---

### 論点3: サンドボックスの優位性

| | エージェントA（技術検証） | エージェントC（反対派） |
|--|------------------------|----------------------|
| 主張 | WSL2 の bubblewrap サンドボックスが最大の技術メリット。自動許可モードが使える | 現環境で問題なく動作中。サンドボックスのために環境を分裂させるのは本末転倒 |
| **判定** | ○ 技術的には正しい | ○ **実益とリスクのバランスではCが妥当** |

**議長メモ**: サンドボックスの自動許可モードは確かに魅力的（bash コマンド毎の承認が不要になる）。しかし、TORASAN の運用では承認フロー自体が安全弁として機能している面もある。

---

### 論点4: 設定・パスの二重管理

| | エージェントB | エージェントC |
|--|-------------|-------------|
| 主張 | symlink や install.sh の WSL 対応で解決可能 | MEMORY.md 分裂、project_registry.json 重複、スキルバージョン不整合が日常化する |
| **判定** | △ 技術的には回避策あり | ○ **運用コストがメリットを上回る** |

**議長メモ**: TORASANは既に複雑なエコシステム（41スキル、16ナレッジ、12ページGUI、Startup自動起動、通知デーモン）。これに WSL レイヤーを追加すると管理コストが跳ね上がる。Cの「複雑性増大 = 致命的」は妥当。

---

## 3. エージェントA（技術検証）の重要な発見

Bでも Cでもなく、Aの技術検証で判明した重要事実:

| 発見 | 詳細 |
|------|------|
| /mnt/c のI/O性能 | シーケンシャル書込が **5.1倍遅い**、小ファイル読込が **2.3倍遅い** |
| 9P プロトコル | WSL2 は Hyper-V VM のため、Windows ファイルへのアクセスに Plan 9 FS を経由 |
| inotify 非対応 | /mnt/c 上のファイル変更を Linux の inotify で検知できない |
| 改行コード | Windows Git (CRLF) vs WSL Git (LF) の混在問題 |

**議長メモ**: TORASAN のプロジェクトファイルは Windows 側にある。WSL から `/mnt/c/` 経由で操作すると、全ての I/O が 2-5倍遅くなる。これだけで「WSL 側に Claude Code を入れて同じプロジェクトを操作する」構成は非効率。

## 5. 議論の結論

### 全会一致で合意した点

1. **Linux ネイティブツール（valgrind, gcov, lcov 等）を WSL にインストールする価値はある**
2. **しかし、それらを使うために Claude Code を WSL に入れる必要はない**
3. **同一リポジトリを2つの Claude Code で同時操作するのは危険**

### 最終結論: **導入しない（現状維持 + 部分活用）**

**理由**: メリット（サンドボックス、並列実行、ネイティブツール）は実在するが、リスク（設定分裂、パス混在、Git競合、複雑性増大）の方が大きい。特に TORASAN は Windows パスに深く最適化されており、WSL レイヤーの追加は投資対効果が悪い。

### 推奨アクション

| 優先度 | アクション | 理由 |
|--------|----------|------|
| **高** | WSL に valgrind, gcov, lcov をインストール | ISO 26262 規格適合。Git Bash から `wsl valgrind ...` で呼び出す |
| **中** | スキルの Bash コマンドに `wsl` 経由の解析オプションを追加 | 既存ワークフローに自然に統合 |
| **低** | 将来 Windows 版 Claude Code のサンドボックスが改善されたら再評価 | 公式ロードマップに "planned" とある |
| **非推奨** | WSL に Claude Code をインストール | 上記リスクが解消されない限り不要 |

## 6. ユーザーへの回答

**Q: WSL に Linux 版 Claude Code を入れたら自由度や効率は上がりますか？**

**A: 理論上は上がる面もありますが、実運用では逆効果になる可能性が高いです。**

- 技術的にはインストール可能で、サンドボックスや Linux ツールの恩恵はある
- しかし TORASAN は Windows パスに深く最適化されており、設定の二重管理・パス混在・Git競合という致命的リスクを抱え込む
- **より良いアプローチ**: WSL に開発ツール（valgrind, gcov 等）だけを入れ、現在の Claude Code から `wsl <command>` で呼び出す構成

**Q: 複数のインスタンスで並列作業は可能ですか？**

**A: 技術的には可能ですが、現在の Task tool（サブエージェント）が既に並列実行を提供しており、2つ目のインスタンスを常駐させるメリットは限定的です。** むしろ Git 競合やファイル衝突のリスクの方が大きいです。
