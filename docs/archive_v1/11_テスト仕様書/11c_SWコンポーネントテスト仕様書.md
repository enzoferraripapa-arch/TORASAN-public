# SWコンポーネントテスト仕様書

## プロジェクト情報
- **プロジェクト名**: 洗濯機モータ制御システム（TORASAN）
- **適用規格**: IEC 60730 クラスB
- **対象マイコン**: RL78/RX
- **テスト責務**: コンポーネントテスト（モジュールレベル安全機能）

---

## 1. テスト方針・使用ツール

### 1.1 テスト目的
コンポーネント（モジュール）レベルでの安全機能テストを実施し、以下を確認する：
- 各安全機能の異常系・境界値での動作
- フォールトインジェクション条件下での検出動作
- 診断結果と安全状態遷移の正確性

### 1.2 テスト方針

| テスト分類 | 方針 | 重要度 |
|-----------|------|--------|
| フォールトインジェクション | CPU/RAM/ROM異常の検出確認 | 高 |
| 境界値テスト | 閾値付近の判定精度確認 | 高 |
| 異常系テスト | センサ断線、信号固着等の検出 | 高 |
| ストレステスト | 連続異常、複数異常同時発生 | 中 |

### 1.3 テスト環境
| 項目 | 内容 |
|------|------|
| テスト基板 | RL78/RX実機評価ボード |
| 診断計測 | オシロスコープ、ロジックアナライザ |
| フォールトインジェクション | デバッガ、テスト治具 |
| 信号模擬 | 信号発生器、DAコンバータ等 |

### 1.4 使用ツール
- **デバッガ**: IAR Embedded Workbench / e² studio
- **FI実行**: GDB, JTAG経由のメモリ改ざん
- **波形計測**: オシロスコープ、ロジックアナライザ
- **自動化**: テストスクリプト（Python等での自動FI実行）

## 3. テストケース

### 3.1 フォールトインジェクション（FI）テスト

#### 概要
MCU内部の異常（CPU故障、RAM破損、ROM改ざん、クロック異常）をシミュレートし、
自己診断機能による検出動作を確認する。

| TC-ID | 対象コンポーネント | 安全機能ID | FI内容 | 期待結果 | 検証方法 |
|-------|------------------|-----------|--------|---------|----------|
| TC-201 | 自己診断コンポーネント | SR-009 | CPUレジスタ固着模擬（テストパターン注入） | DIAG_FAIL フラグ設定 → 安全状態遷移 | UART出力、レジスタ検出率100%確認 |
| TC-202 | 自己診断コンポーネント | SR-010 | RAM 1ビットフリップ模擬（複数箇所） | DIAG_FAIL → 安全状態遷移 | March C検出パターン確認 |
| TC-203 | 自己診断コンポーネント | SR-011 | ROM 1バイト改ざん模擬（複数バイト） | ROM CRC不一致検出 → DIAG_FAIL | CRC値の不一致を確認 |
| TC-204 | 自己診断コンポーネント | SR-012 | MCUクロック周波数10%逸脱模擬 | クロック異常検出 → DIAG_FAIL | スタートアップ時の周波数検出確認 |

#### TC-201 詳細：CPU レジスタ固着テスト

**FI手順:**
1. デバッガでCPUレジスタに0x0000, 0xFFFF, 0xAAAA, 0x5555等を強制設定
2. 自己診断処理を実行
3. テストパターンレジスタが検出結果と比較

**期待結果:**
- 全テストパターンで不一致を検出
- DIAG_FAIL = 1
- 安全状態遷移処理実行
- UART: "DIAG_CPU_FAULT"出力

**検出率基準:**
- 検出率 = 100% （全テストパターン検出）

#### TC-202 詳細：RAM ビットフリップテスト

**FI手順:**
1. デバッガでRAM領域の異なるアドレスにビットフリップを注入
   - アドレス 0x20000000: ビット0をフリップ
   - アドレス 0x20001000: ビット31をフリップ
   - アドレス 0x2000FFFF: 複数ビットフリップ
2. March Cアルゴリズムによるテスト実行
3. 検出結果を確認

**期待結果:**
- ビットフリップが全てMarch Cで検出
- DIAG_FAIL = 1
- UART: "DIAG_RAM_FAULT"出力

**検出率基準:**
- RAM エラー検出率 = 100% （March Cアルゴリズム）

#### TC-203 詳細：ROM 改ざんテスト

**FI手順:**
1. デバッガでROM領域の複数バイトを改ざん
   - プログラムセクション内の1バイト
   - テーブルデータセクション内の複数バイト
2. CRC計算ルーチンを実行
3. 参照CRC値と比較

**期待結果:**
- CRC不一致を検出
- DIAG_FAIL = 1
- 安全状態遷移
- UART: "DIAG_ROM_CRC_MISMATCH"出力

**検出率基準:**
- ROM CRC 検出率 = 100%

#### TC-204 詳細：クロック周波数異常テスト

**FI手順:**
1. スタートアップ時のクロック測定処理に介入
2. 周波数計測結果を意図的に±10%ずらす
3. ウィンドウ比較判定を実施

**期待結果:**
- クロック異常を検出
- DIAG_FAIL = 1
- スタートアップ診断で異常判定

**検出率基準:**
- クロック異常 検出率 = 100% （±10%逸脱）

### 3.2 境界値テスト

#### 概要
各安全機能の判定閾値付近での動作を確認し、判定精度を検証する。

| TC-ID | 対象コンポーネント | 安全機能ID | テスト条件 | 期待結果 | 合否基準 |
|-------|------------------|-----------|----------|---------|----------|
| TC-205 | モータ制御コンポーネント | SR-002 | RPM値: (THRESHOLD-1), THRESHOLD, (THRESHOLD+1) で3パターン検査 | THRESHOLD超過のみ過速度検出 | 3パターン全PASS |
| TC-206 | 電流監視コンポーネント | SR-005 | 電流値: (LIMIT-0.1A), LIMIT, (LIMIT+0.1A) で3パターン検査 | LIMIT超過のみ過電流検出 | 3パターン全PASS |
| TC-207 | 電圧監視コンポーネント | SR-015 | 電圧値: (下限-0.1V), 下限, 上限, (上限+0.1V) で4パターン検査 | ウィンドウ外のみ異常判定 | 4パターン全PASS |

#### TC-205 詳細：過速度判定閾値テスト

**テスト手順:**
1. ホールセンサ信号発生器で回転数を制御
2. 3つの条件で検査：
   - RPM = THRESHOLD - 1 → 検出なし期待
   - RPM = THRESHOLD → 検出なし期待（境界値はセーフサイド）
   - RPM = THRESHOLD + 1 → 過速度検出期待
3. 各条件下で10秒間の安定動作確認

**期待結果:**
- 閾値未満: 過速度フラグ = 0
- 閾値以上: 過速度フラグ = 1 → PWM停止

**判定基準:**
- 3パターン全てで期待通りの判定 → PASS
- 1パターンでも異なる結果 → FAIL

#### TC-206 詳細：過電流判定閾値テスト

**テスト手順:**
1. DAコンバータで電流センサ入力信号を生成
2. 3つの条件で検査：
   - 電流 = LIMIT - 0.1A → 検出なし期待
   - 電流 = LIMIT → 検出なし期待
   - 電流 = LIMIT + 0.1A → 過電流検出期待
3. ADC変換、LPF処理後の判定結果を確認

**期待結果:**
- 閾値未満: 過電流フラグ = 0
- 閾値超過: 過電流フラグ = 1 → ゲート遮断

**判定基準:**
- 3パターン全てで期待通り → PASS

#### TC-207 詳細：電圧監視ウィンドウ比較テスト

**テスト手順:**
1. 可変電源で電圧を制御
2. 4つの条件で検査：
   - V = 下限 - 0.1V → 異常判定期待
   - V = 下限 + 0.1V → 正常判定期待
   - V = 上限 - 0.1V → 正常判定期待
   - V = 上限 + 0.1V → 異常判定期待
3. 各条件で5秒間の継続確認

**期待結果:**
- ウィンドウ内: 電圧異常フラグ = 0 → モータ起動許可
- ウィンドウ外: 電圧異常フラグ = 1 → モータ起動禁止

**判定基準:**
- 4パターン全てで期待通り → PASS

### 3.3 異常系テスト

#### 概要
センサ断線、信号固着などのハードウェア異常が正しく検出・対応されることを確認する。

| TC-ID | 対象コンポーネント | 安全機能ID | 異常条件 | 期待結果 | 検証方法 |
|-------|------------------|-----------|---------|---------|----------|
| TC-208 | モータ制御コンポーネント | SR-001 | ホールセンサ断線（信号入力なし） | センサ異常検出 → DIAG_FAIL → 安全状態遷移 | GPIO入力が0のまま→検出確認 |
| TC-209 | 電流監視コンポーネント | SR-004 | ADC固着（0x000 or 0xFFF） | ADC異常検出 → DIAG_FAIL → 安全状態遷移 | ADC値が固着→検出確認 |
| TC-210 | 蓋監視コンポーネント | SR-007 | 蓋センサ断線（プルアップ固定） | センサ異常検出（安全側） | GPIO入力が1のまま→検出確認 |
| TC-211 | 安全管理コンポーネント | SR-013 | 複数異常同時発生 | 安全状態遷移（最初の異常で即遷移、その後マスク） | 異常優先度確認、競合なし |
| TC-212 | 安全管理コンポーネント | SR-013 | 安全状態中のリカバリ試行 | リカバリ条件満たすまで安全状態を維持 | 再診断実行→異常解除で復帰許可 |

#### TC-208 詳細：ホールセンサ断線テスト

**異常条件:**
1. ホールセンサ信号線をGNDに接続（入力常時LOW）
2. モータ起動コマンド実行
3. ホールセンサ入力監視処理を実行

**期待結果:**
- センサ入力が一定時間変化しない → 断線判定
- DIAG_SENSOR_FAULT フラグ設定
- 安全状態遷移
- UART: "DIAG_SENSOR_OPEN"出力

**検出時間:**
- 入力異常から100ms以内に検出

#### TC-209 詳細：ADC固着テスト

**異常条件:**
1. デバッガでADC値を強制設定 → 0x000（最小値）
2. 電流監視処理を実行
3. ADC値変化を監視

**期待結果:**
- ADC値が一定のまま → 固着判定
- DIAG_ADC_FAULT フラグ設定
- UART: "DIAG_ADC_STUCK"出力

**検出パターン:**
- 0x000 固着 → 検出
- 0xFFF 固着 → 検出

#### TC-210 詳細：蓋センサ断線テスト

**異常条件:**
1. 蓋センサ信号線をプルアップ（入力常時HIGH）
2. 蓋監視処理を実行

**期待結果:**
- センサ入力が常にHIGH → 異常判定
- 蓋オープン判定と同等の安全処理実行
- モータ停止、蓋ロック解除

**安全性:**
- 断線 → 蓋が開いていると判断 → 安全側の動作

#### TC-211 詳細：複数異常同時発生テスト

**異常条件:**
1. 複数の異常を同時に模擬
   - ROM CRC不一致 + ホールセンサ断線
   - ADC固着 + 電圧異常
2. 全診断処理を同時実行

**期待結果:**
- 最初に検出された異常で即座に安全状態遷移
- その後の異常検出がマスクされる（競合なし）
- UART: 最初の異常コードのみ出力

**検証項目:**
- 状態遷移が1回限り
- 異常優先度の階層構造が機能

#### TC-212 詳細：安全状態からのリカバリテスト

**異常条件:**
1. 安全状態へ遷移（TC-208等で発生）
2. センサ異常の原因を解除
3. リカバリ処理を実行

**期待結果:**
- リカバリ条件（異常解除+診断パス）を満たすまで安全状態を維持
- リカバリ実行後に通常状態へ復帰
- UART: "SYSTEM_RECOVERED"出力

**リカバリ条件:**
- 異常が5回連続で検出されない
- 再度スタートアップ診断を実施
- 全診断がPASS

## 6. テスト実施上の注意事項

### 6.1 安全性に関する注意

1. **フォールトインジェクション実施時**
   - MCUが予期しない動作をする可能性がある
   - 実機基板上でのFI実行は必ず安全機能を優先
   - リセット異常時の対応手順を事前に決定

2. **異常系テスト実施時**
   - モータが予期せず動作することがある
   - テスト前に物理的な安全確保（モータキル機構等）
   - 複数異常テストは慎重に実行

### 6.2 テストデータ保管

1. **計測波形の保存**
   - オシロスコープのスクリーンショット
   - CSV形式での生波形データ
   - テストケース毎にフォルダ分類

2. **ログデータ**
   - UART出力ログの全文保存
   - タイムスタンプ付きで記録
   - テスト結果との紐付け

## 注記

本仕様書は洗濯機モータ制御システムのコンポーネントテスト計画を定めたものである。
IEC 60730 クラスB適合性確保のため、特に以下を重視する：

1. **フォールトインジェクション**: MCU内部の異常を網羅的に検出
2. **境界値テスト**: 判定精度の確認
3. **異常系テスト**: フェイルセーフ機能の確実性

テスト実施前に安全責任者による承認を得ること。
テスト中に予期しない不具合が発生した場合は、即座にテストを中止し、
リスク評価を実施した上で対応を決定すること。
