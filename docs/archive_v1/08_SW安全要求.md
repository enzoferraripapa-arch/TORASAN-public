## 2. SW安全要求（SR-XXX）

### 2.1 モータ回転数制御系

#### **[SR-001]** モータ回転数計測処理

- **要求名：** ホールセンサ入力キャプチャによる回転数算出
- **説明：**
  - ホールセンサからのパルス入力をカウント（16ビットCAPレジスタ使用）
  - 計測周期：100ms（10Hz）で回転数[rpm]を算出
  - 計測式：RPM = (PULSE_COUNT / 3) × 60 × 10（3パルス/回転、10Hz計測）
  - 計測値はRAM領域に保存し、保守診断用に記録する

- **上位要件：**
  - TSR-001 ← FSR-001 ← SG-001（モータ回転数監視）

- **下位展開：**
  - SA-001（PH-09：計測誤差分析）
  - UT-001（PH-10：パルスカウント動作確認）
  - TC-001（PH-11：計測値精度テスト）

- **安全クラス：** B
- **検証方法：** 機能テスト
- **ステータス：** 暫定
- **トレース状態：** 🔄展開中

#### **[SR-010]** RAM自己テスト（March C）

- **要求名：** RAM全域の機能テスト
- **説明：**
  - **スタートアップテスト：** 電源投入直後にRAM全域（SRAM 5.5KB=5632B）に対してMarch Cアルゴリズムを実行
    - March C：(↑0); (↑1,0); (↓1,0); (↓0,1) の4フェーズ
    - テストサイズ：容量32KBまで対応、本機は5.5KBのため単一実行
    - テスト時間：500ms程度

  - **ランタイムテスト：** 定期的に分割テストを実行
    - 分割サイズ：1KB×6ブロック（5.5KB）
    - 実行周期：メインループ内で順序実行、全ブロック1周の時間500ms以内
    - テスト詳細度：Reduced March Cアルゴリズム（スタートアップより簡易）

  - テスト結果：status_ram_test フラグに記録
  - FAIL時：エラーコード0x02でリセット要求
  - 規格参照：IEC 60730 Annex H Table H.2

- **上位要件：**
  - TSR-009 ← FSR-007 ← SG-004

- **下位展開：**
  - SA-010（PH-09：March Cアルゴリズムの有効性分析、1周期時間の妥当性）
  - UT-010（PH-10：RAM テスト動作確認）
  - TC-010（PH-11：全アドレスカバレッジテスト）

- **安全クラス：** B
- **検証方法：** 実装レビュー、機能テスト、カバレッジ分析
- **ステータス：** 暫定
- **トレース状態：** 🔄展開中

#### **[SR-012]** クロック監視処理

- **要求名：** 内部クロックと外部基準の相互監視
- **説明：**
  - メインクロック（32MHz）と低速オシレータ（15kHz LOCO）の逸脱を監視
  - 監視方法：
    1. 内部クロック（32MHz MAIN）と内部基準クロック（15kHz LOCO）の相互監視
    2. 同期信号をカウント、目標値との偏差を計算
    3. 逸脱率 = |実測周期 - 期待周期| / 期待周期
  - 検出閾値：±4%（32MHz MAIN vs 15kHz LOCO）
  - 監視周期：1000ms（1秒毎に点検）
  - テスト結果：status_clock フラグに記録
  - FAIL時：エラーコード0x04でリセット要求
  - 規格参照：IEC 60730 Annex H Table H.4

- **上位要件：**
  - TSR-011 ← FSR-007 ← SG-004

- **下位展開：**
  - SA-012（PH-09：許容範囲±4%の妥当性分析）
  - UT-012（PH-10：クロック監視動作確認）
  - TC-012（PH-11：逸脱検出テスト）

- **安全クラス：** B
- **検証方法：** 実装レビュー、機能テスト

#### **[SR-013]** 安全状態遷移処理

- **要求名：** 異常検出時の統一停止処理フロー
- **説明：**
  - SR-009～SR-012 または SR-002/005のいずれかで異常が検出された場合、以下順序で遷移：
    1. **段階1（即座）：** 全PWM出力停止（U/V/W相=Low）→ inverter_stop_flag=1
    2. **段階2（同時）：** 安全状態フラグセット（system_safe_mode=1）→ タイマ停止 → 割込禁止
    3. **段階3（次サイクル）：** UI/表示器へエラー通知（LCD表示 or LED点灯 or ブザー出力）
    4. **段階4（保留）：** 電源リセット待ち（SR-016 WDT タイムアウト）

  - 異常ログ記録：RAM内に異常コード（0x01～0x04）を記録
  - 復帰条件：電源再投入のみ（ソフトリセット不可）

- **上位要件：**
  - TSR-012 ← FSR-008 ← SG-004

- **下位展開：**
  - SA-013（PH-09：遷移順序の妥当性分析、レイテンシ分析）
  - UT-013（PH-10：遷移処理動作確認）
  - TC-013（PH-11：全異常パターン遷移テスト）

- **安全クラス：** B
- **検証方法：** 実装レビュー、機能テスト
- **ステータス：** 暫定
- **トレース状態：** 🔄展開中

### 2.6 ウォッチドッグタイマ系

#### **[SR-016]** WDT管理処理

- **要求名：** ウォッチドッグタイマによる異常リセット
- **説明：**
  - RL78内蔵WDT（12ビット、fINT= 32kHz base）を使用
  - 設定値：カウンタ周期100ms
  - メインループ内で周期的にWDTをキック（カウンタリセット）
    - キック箇所：メインループ100Hz周期の最後（制御周期10ms以内）
  - キック間隔：最大 50ms を超えないこと

  - **異常検出時の動作：**
    1. キック忘れ、CPU制御暴走、無限ループなどでWDTキックが途絶
    2. WDTカウンタがOVERFLOW
    3. CPUリセット信号発生 → チップ全体がリセット → スタートアップから再開（SR-009）

  - テスト結果：WDTトリガは status_wdt_kick フラグで監視
  - 規格参照：IEC 60730 Annex H Table H.1（CPU保護）

- **上位要件：**
  - TSR-015 ← FSR-007/008 ← SG-004

- **下位展開：**
  - SA-016（PH-09：WDT周期の妥当性分析）
  - UT-016（PH-10：WDT動作確認）
  - TC-016（PH-11：キック間隔・タイムアウト測定テスト）

- **安全クラス：** B
- **検証方法：** 機能テスト、実装レビュー
- **ステータス：** 暫定
- **トレース状態：** 🔄展開中

## 3. 診断要求（DR-XXX）

### **[DR-001]** スタートアップ自己診断シーケンス

- **要求名：** 電源投入後の初期検査フロー
- **説明：**
  - 電源投入直後、以下を順次実行：
    1. SR-009（CPUレジスタテスト）→ PASS/FAIL確認
    2. SR-010（RAM March Cテスト）→ PASS/FAIL確認
    3. SR-011（ROM CRC検査）→ PASS/FAIL確認
    4. SR-012（クロック監視初期化）→ 正常性確認

  - **判定：** 全テストPASSでのみ通常動作へ遷移
    - 1つでもFAILなら、SR-013へ → 安全状態、電源リセット待ち

  - 診断カバレッジ（DC）目標：**95%**
    - 対象：CPU/RAM/ROM/Clock脆弱性のうち、本テストで検出可能な範囲

- **検証方法：** スタートアップ時ログ記録、テスト結果フラグ確認
- **ステータス：** 暫定
- **トレース状態：** 🔄展開中

---

### **[DR-002]** ランタイム自己診断シーケンス

- **要求名：** 定期的な分割自己診断
- **説明：**
  - 通常動作中、以下をタスク分割・周期的に実行：
    - **SR-010（RAM分割テスト）：** 512B×16ブロック、メインループ内順序実行
    - **SR-011（ROM分割CRC）：** 4KB×16ブロック、メインループ内順序実行
    - **SR-012（クロック監視）：** 1000ms周期で定期点検

  - 全RAMカバレッジ目標：最大 **500ms以内に1周**
  - 全ROMカバレッジ目標：最大 **5秒以内に1周**

  - FAIL検出時：SR-013へ → 安全状態遷移

  - 診断カバレッジ（DC）目標：**85%**
    - 対象：ランタイムメモリ脆弱性（RAM/ROM脱落検出）

- **検証方法：** ランタイム診断ログ記録、カバレッジ計測
- **ステータス：** 暫定
- **トレース状態：** 🔄展開中

---

### **[DR-003]** ADC診断

- **要求名：** ADC変換結果の異常検出
- **説明：**
  - ADC変換後、各チャネルの出力値（CH0電流、CH1電圧）に対して異常チェック実施：
    - **0値固着検出：** 連続5回で0値が続く → FAIL
    - **フルスケール固着検出：** 連続5回でMax値（0xFFF:12ビット）が続く → FAIL

  - 検出時：フラグ adc_error_flag をセット → SR-013へ

  - 診断カバレッジ（DC）目標：**60%**
    - 検出可能：静的フォルト（オープン、ショート）の一部
    - 未検出：低頻度ノイズ、非線形性（ADC校正値不一致）

- **検証方法：** 機能テスト、異常パターン注入テスト
- **ステータス：** 暫定
- **トレース状態：** 🔄展開中

---

### **[DR-004]** GPIO診断

- **要求名：** 安全関連GPIO（蓋センサ）の固着検出
- **説明：**
  - 蓋センサGPIO（P1_0）に対して定期的に動的テスト実施：
    - **プルアップテスト：** P1_0を入力モード（内部PU有効）→ 値読み出し（High期待）
    - **プルダウンテスト：** GPIO設定を動的に切り替え、Pull-Down状態→値読み出し（Low期待）
    - テスト周期：1000ms

  - **固着判定：**
    - PU時にLow が続く → オープンフォルト（配線断）
    - PD時にHigh が続く → ショートフォルト（GND短絡）

  - 検出時：フラグ gpio_error_flag をセット → SR-013へ

  - 診断カバレッジ（DC）目標：**60%**
    - 検出可能：配線断、常時ショート
    - 未検出：間欠フォルト、信号遅延

- **検証方法：** 実装レビュー、機能テスト、フォルト注入テスト
- **ステータス：** 暫定
- **トレース状態：** 🔄展開中
