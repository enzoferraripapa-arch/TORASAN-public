# 外部ライブラリ管理

**プロジェクト：** 洗濯機モータ制御システム（IEC 60730 クラスB）
**適用規格：** IEC 60730 / IEC 61508 / ISO 26262
**ポリシー：** ベアメタル実装（外部ライブラリなし）

---

## 1. ライブラリ管理ポリシー

### 1.1 基本方針

本プロジェクトは **ベアメタル実装** を採用し、外部ライブラリ（RTOS、ミドルウェア、オープンソース）の使用を **最小化** します。

| 決定 | 理由 |
|:--|:--|
| **RTOS不使用** | 超小規模 MCU（32KB RAM）、スーパーループで十分、RTOSの認証取得コスト |
| **ハードウェア抽象化層は自作** | MCU固有の最適化、セキュリティ監視、レイテンシ確保 |
| **数学ライブラリなし** | 安全コードで浮動小数点禁止、固定小数点で自作 |

### 1.2 ライブラリ追加の承認プロセス

将来的に外部ライブラリの採用を検討する場合は、以下の条件をすべて満たす必要があります：

#### 要件1：機能安全対応
- [ ] IEC 60730 / IEC 61508 / ISO 26262 対応済み
- [ ] SEooC（Safety Element out of Context）安全論証の入手
- [ ] 安全認証レベル（ASIL/SIL）の取得
- [ ] 認証ドキュメントのレビュー完了

#### 要件2：ライセンス確認
- [ ] オープンソースライセンス（GPL等）の確認
- [ ] 法務・知財チームのレビュー
- [ ] コンプライアンス教育の実施

#### 要件3：セキュリティ監査
- [ ] CVE（Common Vulnerabilities and Exposures）データベース確認
- [ ] 既知の脆弱性がないこと
- [ ] ソースコード入手可否の確認
- [ ] コード品質・保守性評価

#### 要件4：プロジェクト適合性
- [ ] MCU リソース（メモリ、CPU）への適合
- [ ] リアルタイム性への影響評価
- [ ] テスト可能性の確認
- [ ] 保守性への影響評価

### 1.3 ライブラリ非使用宣言

本プロジェクト開発の現段階では、以下の理由により外部ライブラリを使用しません：

| カテゴリ | 理由 | 代替実装 |
|:--|:--|:--|
| **RTOS** | MCU性能（32KB RAM）では割に合わない、スーパーループで十分 | スーパーループ（1ms周期） |
| **HAL/ドライバ層** | MCU固有の最適化が必須、セキュリティ監視が困難 | カスタム HAL（hal_*.c） |
| **暗号化/CRC** | 小規模実装（CRC32のみ必要）、外部依存排除 | safe_crc32_compute() |
| **浮動小数点演算** | 安全コードでは禁止、不要 | 固定小数点演算（16.16形式） |
| **デバッグトレース** | 本物のシリアル UART で十分 | hal_uart.c |

## 6. 代替実装例

### 6.1 CRC32 計算

外部ライブラリの代わりに以下の自作実装を使用：

```c
/**
 * CRC-32 計算ルーチン
 * 多項式: 0x04C11DB7
 * 初期値: 0xFFFFFFFF
 * 最終XOR: 0xFFFFFFFF
 */
uint32_t crc32_compute(const uint8_t *data, uint32_t length) {
    uint32_t crc = 0xFFFFFFFFUL;

    for (uint32_t i = 0; i < length; i++) {
        uint8_t byte = data[i];
        crc ^= byte;

        for (uint8_t j = 0; j < 8; j++) {
            if (crc & 0x80000000UL) {
                crc = (crc << 1) ^ 0x04C11DB7UL;
            } else {
                crc = crc << 1;
            }
            crc &= 0xFFFFFFFFUL;
        }
    }

    return crc ^ 0xFFFFFFFFUL;
}
```

### 6.2 タイマ管理

```c
/**
 * スーパーループ用タイマ管理
 * RTOS不要
 */
typedef struct {
    uint32_t tick_ms;      // 現在の時刻（ms）
    uint32_t last_task_10ms;  // 前回実行時刻
    uint32_t last_task_100ms;
    uint32_t last_task_1000ms;
} TimerManager_t;

void TimerMgr_Init(TimerManager_t *mgr) {
    mgr->tick_ms = 0;
    mgr->last_task_10ms = 0;
    mgr->last_task_100ms = 0;
    mgr->last_task_1000ms = 0;
}

uint8_t TimerMgr_IsTime_10ms(TimerManager_t *mgr) {
    if ((mgr->tick_ms - mgr->last_task_10ms) >= 10) {
        mgr->last_task_10ms = mgr->tick_ms;
        return 1;
    }
    return 0;
}

// メインループで呼び出し
void main_loop() {
    while (1) {
        TimerMgr_IncrementTick();  // タイマ割込みで増加

        if (TimerMgr_IsTime_10ms(&timer_mgr)) {
            motor_control_task();
        }
        if (TimerMgr_IsTime_100ms(&timer_mgr)) {
            diagnostic_task();
        }
    }
}
```

## 8. FAQ

### Q1: なぜ RTOS を使わないのか？
**A:**
- MCU メモリが小さい（RAM 32KB）
- スーパーループで十分な周期（1ms～1000ms）
- RTOS 自体の安全認証取得コストが大きい
- ベアメタル実装により制御レイテンシが確定可能

### Q2: ライブラリ追加時、どのくらい時間がかかる？
**A:** 典型的には **3-4週間**
- 技術評価: 1週間
- 安全評価: 1-2週間
- 法務評価: 1週間
- 統合テスト: 1-2週間（並行可）

### Q3: マイナーバージョンアップ（X.Y.1 → X.Y.2）は承認が必要？
**A:** セキュリティパッチの場合は **優先度高で事前承認**。
バグ修正のみなら関係なければ事後報告でも可。
ただしプロジェクト安定期は **固定化推奨**。

### Q4: オープンソース（GPL）ライブラリは使えない？
**A:** GPL は **開示義務**があるため、クローズドソース製品なら原則不可。
商用ライセンス版や、プロジェクト自体がオープンソースなら検討可。

### Q5: 自作ライブラリは評価対象？
**A:** **対象外**。ただし第三者が開発したコンポーネント扱いなら**検討**。
内部チーム開発は通常のコード品質レビューで十分。
