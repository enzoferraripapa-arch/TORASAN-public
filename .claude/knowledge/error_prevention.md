# 凡ミス防止ガイド

12セッション分の振り返りデータから抽出した再発パターンと防止策。

## 1. 再発パターンと対策

### A. パス・ファイル操作ミス（発生率: 5/12セッション）

| パターン | 原因 | 防止策 |
|---------|------|--------|
| メモリパス slug 推測ミス | 手動算出で `_` → `-` 変換等の誤り | **必ず `ls` で実在確認してから使う**。推測禁止 |
| WT vs メインリポ混同 | 作業ディレクトリの曖昧さ | Edit 前に `$PWD` を確認。WT 内か否か意識 |
| ファイル検索対象の誤特定 | 類似ファイル名（MEMORY.md vs CLAUDE.md） | Grep で対象テキストを含むファイルを特定してから Edit |

### B. Read → Edit 前提条件違反（発生率: 3/12セッション）

| パターン | 原因 | 防止策 |
|---------|------|--------|
| サブエージェントの Read を自分の Read と誤認 | ツール制約の理解不足 | **メインエージェントで必ず Read してから Edit** |
| 並列 Read+Edit で競合 | 完了順序が不定 | 10ファイル超は Python スクリプトで一括変更 |
| Read していないファイルを Edit | 急いで作業開始 | Edit ツール実行前に「このファイルを Read したか？」を自問 |

### C. セッション引き継ぎミス（発生率: 3/12セッション）

| パターン | 原因 | 防止策 |
|---------|------|--------|
| session_state.md 検出失敗 | パス slug の推測ミス | Phase B で `ls ~/.claude/projects/` 実行を必須化 |
| 未コミットファイル見落とし | git status 未確認 | session start 時に git status を必ず表示 |
| 前セッション改善項目の未対応 | フォローアップ不足 | session_state.md の unresolved_retro_items を Phase C で表示 |

### D. 確認・検証漏れ（発生率: 4/12セッション）

| パターン | 原因 | 防止策 |
|---------|------|--------|
| 規格誤適用（ASIL on IEC 60730） | product_override 未確認 | `/select-standard check` を Phase 開始時に実行 |
| 表記レベル不統一 | レビューチェックリスト不在 | 成果物作成後にマルチエージェントレビュー必須 |
| セキュリティ問題の後検出 | レビュータイミングが遅い | 各機能完了直後にセキュリティチェック実施 |

### F. SPICE WP 修正時の波及漏れ（発生率: 1/12セッション）

| パターン | 原因 | 防止策 |
|---------|------|--------|
| WP テーブル部分修正（4/7のみ修正） | SYS.1↔SYS.2↔SYS.3 の依存関係の理解不足 | **WP 変更時は関連プロセスを横断チェック**（§7 チェックリスト使用） |
| 修正漏れのレビュー後検出 | 自己チェックなしで Edit 開始 | Edit 前に PROCESS.md §2.3 WP テーブル全体を Read して全エントリ確認 |

### E. 環境固有の問題（発生率: 2/12セッション）

| パターン | 原因 | 防止策 |
|---------|------|--------|
| cmd.exe で外部ツールが見つからない | cmd.exe と Git Bash で PATH が異なる | Bash ツール実行時に `which` で確認。見つからなければ Git Bash 経由を試す |
| 新規ツール導入後に環境チェックで検出失敗 | KNOWN_TOOL_PATHS 未登録 | **ツール追加時フロー**（下記）に従い KNOWN_TOOL_PATHS + env_state.md を同時更新 |
| 大規模変更でコンテキスト消費過多 | 一括作業でトークン枯渇 | **3ファイル超は Plan Mode**。/compact + サブエージェント委任 |
| Git Bash で /tmp/ にファイル書き出し | /tmp/ は MSYS2 仮想パスで Windows アプリから参照不可 | **一時ファイルは `$HOME/` 配下を使用**。/tmp/ 禁止 |

**ツール追加時の基本フロー:**

```
1. cmd.exe で見つかるか確認: cmd.exe //c "where {tool}"
2. 見つからない場合:
   (a) Git Bash で which {tool} → 実パスを特定
   (b) environmentService.ts の KNOWN_TOOL_PATHS にエントリ追加
   (c) TOOL_CHECKS に未登録なら併せて追加
3. ビルド確認: npx tsc --noEmit
4. preview_start → 環境管理ページで OK 表示を確認
5. env_state.md の環境スタック + 変更ログを更新
6. 仕様書再生成: python scripts/generate_manuals.py
```

### G. 外部ツール・プラグイン導入ミス（発生率: 1/12セッション）

| パターン | 原因 | 防止策 |
|---------|------|--------|
| プラグインディレクトリ名 ≠ manifest ID | GitHub リポ名とプラグイン ID が異なる | **manifest.json の `id` を確認してからディレクトリ命名** |
| 一律スケール適用で品質悪化 | 全図に同じ係数を適用 | **根本解決優先: 図ごとに明示サイズを設定**。ワークアラウンド禁止 |
| 大量ファイル変換でコンテキスト枯渇 | 直列処理で全件処理 | **並列エージェント3分割**で処理（6件以上の場合） |

## 2. 事前マルチエージェント議論（全タスク必須）

**全てのユーザー指示に対し、実行前に複数エージェントで議論する。**

### 2.0 PDCA タスク実行フロー（全指示・省略不可）

```
ユーザーから指示を受ける
  ↓
【Plan: 議論エージェント起動】
  Agent-A（実行計画）: 何をどの順序で変更するか、影響範囲の列挙
  Agent-B（リスク分析）: 見落とし・副作用・波及先の指摘
  ※ 2エージェント並行、haiku モデル
  ↓
【Plan: 議論結果の統合】
  メインエージェントが両者の結果を統合し、実行計画を確定
  矛盾がある場合はメインエージェントが判断
  ↓
【Do: 実行】
  確定した計画に従って実施
  ↓
【Check: 検証】
  実行結果が計画と一致しているか確認:
  □ 計画した変更が全て適用されたか
  □ 意図しない変更が混入していないか
  □ Agent-B が指摘したリスクが顕在化していないか
  ※ メインエージェント自身が実施（追加エージェント不要）
  ↓
【Act: 改善エージェント起動】
  Agent-C（改善提案）: 実行結果と検証結果を受け、改善策を提案
  Agent-D（プロセス改善）: 今回の作業プロセス自体の改善点を指摘
  ※ 2エージェント並行、haiku モデル
  → メインエージェントが統合し、以下を実施:
  □ 問題検出時: 即座に修正 or 修正計画を立てて再実行
  □ 再発防止パターンを発見 → §1 に追記
  □ プロセス改善の余地を発見 → MEMORY.md に記録
  ↓
【事後レビューゲート】
  3ファイル以上変更した場合 → 追加で整合性レビュー（§2.1）
```

### 2.0.1 議論エージェントのプロンプト

```
Agent-A（実行計画）:
「ユーザー指示: {instruction}
 以下を報告せよ:
 1. 変更対象ファイルの一覧（Grep/Glob で実際に検索して特定）
 2. 推奨する実行順序
 3. 各ファイルの具体的な変更内容（何を何に変えるか）
 結果: ファイル一覧 + 実行順序 + 変更内容」

Agent-B（リスク分析）:
「ユーザー指示: {instruction}
 以下を報告せよ:
 1. Agent-A が見落としそうな波及先（Grep で関連参照を検索）
 2. 変更による副作用リスク（壊れるテスト、整合性の崩れ等）
 3. 確認すべき前提条件（ファイルの現状確認が必要な箇所）
 結果: 波及先 + リスク + 前提条件」

Agent-C（改善提案）[Act フェーズ]:
「実行結果: {execution_summary}
 検証結果: {check_result}
 以下を報告せよ:
 1. 検証で検出された問題の修正案（具体的な修正内容）
 2. 変更品質の評価（意図通りか、副作用はないか）
 3. 再発防止策（§1 に追記すべきパターンがあれば提案）
 結果: 修正案 + 品質評価 + 再発防止策」

Agent-D（プロセス改善）[Act フェーズ]:
「実行結果: {execution_summary}
 検証結果: {check_result}
 以下を報告せよ:
 1. 今回の作業プロセスで非効率だった点
 2. 次回以降に改善すべきワークフロー（MEMORY.md 記録候補）
 3. エージェント議論（Plan）の精度評価（見落としがあったか）
 結果: 非効率点 + ワークフロー改善 + 議論精度評価」
```

### 2.1 事後レビューゲート（3ファイル以上変更時）

3ファイル以上を変更した場合、実行完了後に追加レビューを実施:

```
□ 変更ファイル間の相互参照が正しいか
□ 変更に伴い更新が必要だが未更新のファイルがないか
□ 全変更が当初の議論結果と一致しているか
```

## 3. 「3回失敗ルール」

同じアプローチで3回失敗したら:
1. 即座に立ち止まる
2. 別のアプローチを検討（マルチエージェントで複数案を並行検討）
3. それでも解決しない場合はユーザーに報告

`preview_start` はポート 3000 の既存プロセス（Startup VBS）を kill する。
検証後に復旧しないとユーザーがダッシュボード（localhost:3000）を使えなくなる。

```
【検証前】
□ preview_start で既存 Startup プロセスが停止することを認識

【検証後】
□ preview_stop でプレビューサーバーを停止
□ アプリを再起動: cd app && npm run dev（バックグラウンド）
  または: wscript app/start_torasan_app.vbs
□ localhost:3000 でダッシュボードが表示されることを確認
```

## 7. SPICE WP 修正時チェックリスト

PROCESS.md の WP テーブル（§2.3）を修正する場合、以下を必ず実施する。

```
【修正前】
□ PROCESS.md §2.3 WP テーブル全体を Read（部分読み込み禁止）
□ 修正対象 WP の関連プロセスを特定（依存マップ参照）

□ マルチエージェントレビュー必須（§2 条件4 に該当）
```

## 8. Hook デバッグ手順（S30 教訓）

hook が動作しない場合の調査手順。推測での原因特定を禁止し、ログ確認を最優先する。

```
【Step 1: イベント発火確認（最優先）】
□ ~/.claude/debug/*.txt でイベント名を grep
  grep -r "Stop\|Notification\|PreCompact" ~/.claude/debug/
  → 該当イベントが記録されていなければ「イベント自体が未発火」
  → 記録されていればフック実行側（スクリプト）の問題

【Step 2: 環境確認】
□ 実行環境を特定（CLI / Claude Desktop / Agent SDK）
  → Notification イベントは CLI ターミナル直接実行時のみ発火
  → Claude Desktop / Agent SDK 環境では Stop hook で代替
□ 設定変更はセッション再起動後に有効（セッション中は反映されない）

【Step 3: 設定マージ確認】
□ グローバル（~/.claude/settings.json）とプロジェクト（.claude/settings.json）の
  両方に同一 hook がないか確認 → 重複は削除
□ プロジェクト settings.json は追加設定のみ記載（グローバルと重複させない）
```

## 9. project.json 同時書き込み防止（S31 調査）

project.json は 800行超の大型 JSON。Edit ツールに file lock がないため、並行書き込みで先祖返りするリスクがある。

```
【運用ルール】
□ WSL デーモン（wsl-spice-improver.py）実行中は project.json 変更スキルを実行しない
□ project.json を変更するスキル（manage-tbd, update-record, commit-change 等）は直列実行
□ WSL エージェントのプロンプトに「Do NOT edit project.json」を明示（既に実装済み）

【競合が疑われる場合の確認手順】
□ git diff project.json で直近の変更差分を確認
□ git log --oneline -5 project.json で更新履歴を確認
□ 意図しないセクション消失があれば git show HEAD~1:project.json で復元
```

## 10. 大量タスク中間 compact ガイドライン

コンテキスト枯渇を防ぐための中間ポイント判定基準。

```
【判定基準 — いずれかに該当したら /compact を検討】
□ 3ファイル超を連続編集した
□ 3コミット超を1セッション内で実行した
□ サブエージェントの結果が大量（長文レスポンス3回以上）
□ 体感でセッション後半に入った（作業量の半分超を消化）

【/compact 実行手順】
1. 現在の作業状態を session_state.md に書き出す
2. /compact を実行（要約: 現在のフェーズ + 残タスク + 変更済みファイル）
3. compact 後に session_state.md から状態を復元して作業再開

【セッション分割判定 — /compact では不十分な場合】
□ 5ファイル超のリファクタ → 独立セッションに分割
□ 異なるスコープの作業が混在 → スコープ単位でセッション分割
```
