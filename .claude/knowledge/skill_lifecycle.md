# スキルライフサイクル管理ナレッジベース

本ドキュメントはスキル（カスタムスラッシュコマンド）のライフサイクル管理プロセス、
成熟モデル、レビュー基準、および改善サイクルの運用知識をまとめたものである。

---

## 1. スキルライフサイクルモデル

### 1.1 ライフサイクルステージ

```
  Draft ──→ Active ──→ Practiced ──→ Mature ──→ Optimized
    │          │           │            │           │
    │          │           │            │           ▼
    │          │           │            │       (継続改善)
    │          │           │            │
    ▼          ▼           ▼            ▼
 Deprecated  Deprecated  Deprecated  Deprecated
```

| ステージ | 成熟度 | 定義 | 遷移条件 |
|---------|--------|------|---------|
| Draft | L1 | 基本手順が定義されている | .md ファイルが存在する |
| Active | L2 | ナレッジベース参照あり | `## ナレッジベース参照` セクションが存在 |
| Practiced | L3 | 実行経験あり | feedback が1回以上記録されている |
| Mature | L4 | フィードバック反映済み | improve 1回以上 + 平均評価 4.0 以上 |
| Optimized | L5 | 継続改善確立 | improve 2回以上 + 平均評価 4.5 以上 + ナレッジ更新あり |
| Deprecated | — | 廃止・統合済み | 後継スキルに置き換え or 不要と判断 |

### 1.2 遷移ルール

- **昇格**: 上位ステージへの遷移は客観的データ（フィードバック記録）に基づく
- **降格**: 基本的に発生しない。重大な問題がある場合のみ Active に戻す
- **廃止**: 統合・分割・不要化の場合。Deprecated にしてから一定期間後に削除

## 2. スキル品質基準

### 2.1 構造要件（静的品質）

全スキルが満たすべき構造:

```markdown
# /{skill-name} — {タイトル}

{1行説明}

引数: $ARGUMENTS（...）

## 手順
### {mode} モード — {説明}
#### Step N: {手順名}

## 関連スキル

## 注意事項

## ナレッジベース参照
```

| セクション | 必須 | 品質基準 |
|-----------|------|---------|
| タイトル行 | ✓ | `# /{name} — {説明}` 形式 |
| 1行説明 | ✓ | 何をするスキルか1文で記述 |
| 引数定義 | ✓ | $ARGUMENTS のパース規則を明記 |
| 手順 | ✓ | Step 単位で分解、具体的な操作を記述 |
| 出力フォーマット | 推奨 | 出力例を code block で提示 |
| 関連スキル | ✓ | 関連スキルとの連携を明記 |
| 注意事項 | ✓ | 制約・前提条件・安全策を明記 |
| ナレッジベース参照 | ✓(L2+) | 参照するナレッジファイルとその内容を列挙 |

### 2.2 品質レベル定義

| レベル | 名称 | 条件 |
|--------|------|------|
| A (完備) | 全セクション完備 + ナレッジ参照あり | 構造100% + L2以上 |
| B (良好) | 主要セクション完備 | 手順 + 関連 + 注意事項あり |
| C (基本) | 最低限の手順のみ | 手順セクションのみ |
| D (不足) | 必須セクション欠落 | タイトル or 手順なし |

### 2.3 UX設計原則

スキル設計時に遵守すべきユーザー体験の原則:

| # | 原則 | 説明 | 違反例 |
|---|------|------|--------|
| UX-1 | **ワークフロー分岐はユーザー確認必須** | プロジェクト選択・モード選択など作業方向を決める分岐は、自動判定せず必ずユーザーに聞く | project.json の有無で勝手に新規/既存を判定 |
| UX-2 | **複数候補の提示** | 選択肢が複数ありうる場合は一覧を提示して選ばせる。1件でも確認する | 既存PJが1つだから自動選択 |
| UX-3 | **裏の探索 + 表の確認** | 情報収集は裏で行い、結果をもとに選択肢を構成してユーザーに提示する | 探索結果をそのまま判定に使い確認なしで進む |

## 3. フィードバック収集プロセス

### 3.1 収集タイミング

- **最適**: スキル実行直後（記憶が新鮮なうちに）
- **許容**: セッション終了前にまとめて（/session end と連携）
- **非推奨**: 次セッション以降（コンテキスト喪失）

### 3.2 フィードバック記録フォーマット

```markdown
## {date} — /{skill-name} 実行フィードバック

- **評価**: {1-5}/5
- **良かった点**: {そのまま維持すべき内容}
- **不足点**: {手順に抜けがあった、出力が不十分等}
- **ナレッジ過不足**: {参照情報が足りなかった、古かった等}
- **問題**: {環境依存、前提条件の不備等}
- **改善提案**: {具体的な改善案}
```

### 3.3 評価基準

| 評価 | 意味 | 基準 |
|------|------|------|
| 5 | 完璧 | 手順通りに実行でき、出力も十分。改善不要 |
| 4 | 良好 | 概ね問題なし。軽微な改善の余地あり |
| 3 | 普通 | 使えるが不足感あり。手動補完が必要だった |
| 2 | 不十分 | 手順に大きな抜け。追加の調査・作業が必要だった |
| 1 | 不適合 | 手順が実行不能 or 誤り。根本的な見直し必要 |

## 4. 改善分析プロセス

### 4.1 分析カテゴリ

| カテゴリ | 説明 | 改善先 |
|---------|------|--------|
| 手順不足 | Stepの抜け、順序の問題 | skill .md の手順セクション |
| ナレッジ不足 | 参照情報の不足・陳腐化 | knowledge/ ファイル |
| 出力不足 | 出力フォーマットが見づらい・不十分 | skill .md の出力セクション |
| 前提条件 | 必要な前提の明記漏れ | skill .md の注意事項 |
| 環境依存 | 特定環境でのみ発生する問題 | knowledge/ or skill .md |
| 連携不足 | 他スキルとの連携が不十分 | skill .md の関連スキル |

### 4.2 優先度判定

```
改善優先度 = 頻度 × 影響度

頻度: フィードバックでの出現回数
影響度:
  High — スキルの主要機能に影響（実行不能レベル）
  Medium — 補助的な機能に影響（手動補完で対処可能）
  Low — 見た目・利便性の問題（使えるが不便）
```

## 5. 改善適用プロセス

### 5.1 改善の原則

1. **ユーザー確認必須**: 改善案を提示し、承認を得てから適用
2. **差分最小化**: 必要な箇所のみ変更。構造の大幅改変は避ける
3. **後方互換**: 既存の使い方を壊さない
4. **記録**: 変更内容をコミットメッセージに明記

### 5.2 改善対象の分類

| 対象 | ツール | 注意点 |
|------|--------|--------|
| スキル定義 (.md) | Edit | 手順の追加・修正。構造テンプレートに準拠 |
| ナレッジベース | Edit | 知見の追加。既存内容との整合性確認 |
| project.json | — | スキル改善では通常変更不要 |

### 5.3 改善後の検証

- 変更したスキルの構造が品質基準 (§2.1) を満たすか
- ナレッジベース参照が有効か（ファイルが存在するか）
- 関連スキルとの整合性（参照先のスキルも更新が必要か）

## 7. スキル間連携パターン

### 7.1 連携の種類

| パターン | 説明 | 例 |
|---------|------|-----|
| 前工程→後工程 | あるスキルの出力が別スキルの入力 | /system-design → /sw-design |
| 検証→修正 | 検証スキルが問題検出し修正スキルへ | /health-check → /problem-resolve |
| 実行→記録 | 実行スキルの結果を記録スキルで保存 | /execute-phase → /update-record |
| メタ→対象 | メタスキルが対象スキルを管理 | /skill-manage → 全スキル |
| 環境→実行 | 環境スキルで前提確認後に実行 | /env-check → /static-analysis |

### 7.2 推奨ワークフロー

```
セッション開始
  → /session start（前回状態復元 + 環境チェック）
  → /dashboard（状況把握）
  → /execute-phase（作業実行）
  → /commit-change（成果保存）
  → /session end（反省会 + 状態保存 + master マージ）
  → /skill-evolve feedback（フィードバック記録）
セッション終了
```

## 8. レビュースケジュール

### 8.1 定期レビュー

| 頻度 | 対象 | アクション |
|------|------|----------|
| セッション毎 | 使用したスキル | /skill-evolve feedback でフィードバック記録 |
| 5セッション毎 | 全スキル | /skill-manage audit で品質監査 |
| 改善適用後 | 対象スキル | 次回実行時に効果検証 |

### 8.2 成熟度レビュー基準

| 項目 | 基準 |
|------|------|
| L1→L2 昇格 | ナレッジベース参照セクションを追加 |
| L2→L3 昇格 | 実行フィードバックが1件以上記録 |
| L3→L4 昇格 | improve 1回以上実施 + 平均評価 4.0以上 |
| L4→L5 昇格 | improve 2回以上 + 平均評価 4.5以上 + ナレッジ更新あり |

## 9. スキル廃止プロセス

### 9.1 廃止の判断基準

- 他のスキルに機能が統合された
- プロジェクトの方針変更で不要になった
- 長期間使用されず、ナレッジも陳腐化している

### 9.2 廃止手順

1. Deprecated マークを付与（タイトル行に `[DEPRECATED]` を追加）
2. 後継スキルへの誘導を注意事項に記載
3. 1〜2セッション後に .md ファイルを削除
4. skill_feedback_log.md の該当エントリに廃止記録を追加

## 10. フィードバックログ管理

### 10.1 ファイル構成

```
.claude/knowledge/skill_feedback_log.md
```

### 10.2 ログ構造

```markdown
# スキルフィードバックログ

## 成熟度サマリ
| スキル | レベル | 実行回数 | 平均評価 | 最終改善日 |
|--------|-------|---------|---------|-----------|

## フィードバック履歴
### /{skill-name}
#### {date} — 実行フィードバック
- 評価: {1-5}/5
- ...
```

### 10.3 運用ルール

- ログは `.claude/knowledge/` に配置（Git 管理対象）
- 成熟度サマリは /skill-evolve maturity が参照・更新
- 古いフィードバック（6ヶ月以上）はアーカイブセクションへ移動可
- ログ肥大化防止: 同一スキルの同一問題は集約して記録
