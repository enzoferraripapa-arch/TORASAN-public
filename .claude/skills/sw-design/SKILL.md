# /sw-design — ソフトウェア設計方法論

PH-09（SWアーキテクチャ設計）・PH-10（SWユニット設計）を支援する SW 設計専門スキル。
層構造設計・状態遷移設計・モジュール分割・設計パターン適用・設計レビューを行う。

引数: $ARGUMENTS（オプション。"architecture" | "state-machine" | "module" | "pattern" | "detailed" | "review" | "all" → 省略時は "all"）

## 手順

### architecture モード — アーキテクチャ設計
1. 3層アーキテクチャの構成要素を定義:

| 層 | モジュール | SA-ID | 依存先 | 責務 |
|----|----------|-------|-------|------|
| APP | motor_ctrl | SA-001 | BSW, HAL | モータ速度制御、過速度検知 |
| APP | current_mon | SA-002 | HAL(adc) | 電流監視、過電流判定 |
| APP | lid_mon | SA-003 | HAL(gpio) | 蓋開閉検知 |
| APP | voltage_mon | SA-004 | HAL(adc) | 電圧監視、窓比較 |
| BSW | safety_mgr | SA-010 | dem, HAL | 安全状態管理、遷移制御 |
| BSW | safety_diag | SA-011 | HAL | 自己診断（RAM/ROM/CPU/CLK） |
| BSW | dem | SA-012 | — | 診断イベント記録 |
| BSW | scheduler | SA-013 | HAL(timer) | 周期タスクスケジューリング |
| HAL | hal_* | SA-020〜 | HW | ペリフェラル抽象化 |

2. 依存関係ルール:
   - APP → BSW, HAL（OK）
   - BSW → HAL（OK）
   - HAL → HW（OK）
   - 逆方向依存禁止（コールバックを使う場合は関数ポインタで）
   - 同層内の循環依存禁止

3. docs/09_sw_architecture.md に出力

### state-machine モード — 状態遷移設計
システム全体の安全状態遷移図:

```
                    ┌──────────┐
           電源ON → │  INIT    │ 自己診断実行中
                    └────┬─────┘
                    診断PASS│       診断FAIL
                         ▼          ▼
                    ┌──────────┐  ┌──────────┐
                    │  IDLE    │  │  FAULT   │
                    │  待機    │  │  故障    │
                    └────┬─────┘  └──────────┘
                  運転指令│              ▲
                         ▼              │ 異常検出
                    ┌──────────┐         │
               ┌───▶│ RUNNING  │─────────┘
               │    │  運転中   │
               │    └────┬─────┘
               │    停止指令│
               │         ▼
               │    ┌──────────┐
               └────│ STOPPING │
                    │  停止中   │
                    └────┬─────┘
                    停止完了│
                         ▼
                    ┌──────────┐
                    │  IDLE    │
                    └──────────┘
```

各状態について:
- エントリアクション / エグジットアクション
- 許可されるイベント
- ガード条件
- 遷移先と遷移アクション

安全状態(FAULT)への遷移条件:
- 過速度検知 (>1500rpm)
- 過電流検知 (>8A)
- 蓋開放検知
- WDT タイムアウト
- 自己診断 FAIL（RAM/ROM/CPU/CLK/電圧）

### module モード — モジュール分割設計
各モジュールについて:
1. **責務**: 単一責任の原則に基づく役割定義
2. **公開IF**: ヘッダファイルに公開する関数・マクロ
3. **内部IF**: static 関数・変数（外部から不可視）
4. **データフロー**: 入力データ→処理→出力データ
5. **リソース使用**: RAM(static変数)・Flash(コードサイズ)の見積もり
6. **SR マッピング**: このモジュールが実現する SW安全要求

### pattern モード — 設計パターン適用
組込み安全ソフトウェアに適した設計パターン:

| パターン | 適用先 | 目的 |
|---------|-------|------|
| Watchdog Pattern | scheduler + wdt | メインループ監視 |
| Voter/Comparator | sensor inputs | 入力値の多数決・クロスチェック |
| Safe State Pattern | safety_mgr | 異常時の確定的安全遷移 |
| Cyclic Executive | scheduler | 確定的タスクスケジューリング |
| Defensive Copy | shared data | データ破損検出（ダブルバッファ） |
| Program Flow Monitor | safety_diag | 制御フロー逸脱検出 |

### detailed モード — ユニット詳細設計（PH-10）
各ユニット(UT-XXX)について:
1. アルゴリズム記述（疑似コードまたはフローチャート）
2. 入出力仕様（型、範囲、単位、精度）
3. エラーハンドリング
4. 前提条件・事後条件
5. 計算量・実行時間の見積もり

### review モード — 設計レビュー
チェック項目:
1. **層構造違反**: 逆方向依存・層スキップがないか
2. **モジュール結合度**: 結合度が高すぎるモジュールの特定
3. **安全要求カバレッジ**: 全SR が少なくとも1つのモジュールに割当済か
4. **リソース見積もり**: Flash 64KB / RAM 5.5KB に収まるか
5. **テスタビリティ**: 各モジュールが独立にテスト可能か
6. **MISRA C 準拠性**: §5 型ルールへの設計時点での配慮

## Error Handling（エラー時の対応）

| 障害 | 検出方法 | 復旧手順 |
|------|---------|---------|
| 入力データ不足 | 必要ファイル/キーの不在 | 不足項目を報告し、前提スキルの実行を促す |
| 処理中断 | 予期しないエラー | エラー内容を報告。変更途中のファイルがあれば `git checkout` で復元 |

## 注意事項
- 層間依存は上→下のみ（違反は ERROR）
- 割り込みコンテキストと通常コンテキストのデータ共有は volatile + クリティカルセクション
- グローバル変数は最小限（モジュール内 static を優先）
- 関数の循環的複雑度は 10 以下を目標
- SA-ID の採番は連番を維持
