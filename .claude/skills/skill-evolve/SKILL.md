# /skill-evolve — スキル成熟エンジン

スキル実行後のフィードバックを収集・分析し、スキル定義とナレッジベースを改善する。

引数: $ARGUMENTS（"feedback {skill-name}" = 実行フィードバック記録 | "analyze {skill-name}" = 改善分析 | "improve {skill-name}" = 改善適用 | "maturity" = 全スキル成熟度一覧 | "cycle" = 成熟サイクル実行 | "tech-refresh {skill-name|all}" = 技術最新化 → 省略時は "maturity"）

## Prerequisites（前提条件）

本スキル実行前に以下を確認:

| # | 条件 | 確認方法 | 未充足時 |
|---|------|---------|---------|
| 1 | Git リポジトリ内で実行 | `git status` | リポジトリ外では実行不可 |

## 目的

スキルは「作って終わり」ではなく、**使って → 振り返って → 改善して → また使う** のサイクルで成熟させる。本スキルはこの PDCA サイクルを駆動するエンジンである。

```
    ┌─────────┐
    │ 実行    │ ← /execute-phase, /fmea, /test-design 等
    └────┬────┘
         │ 結果・気づき
    ┌────▼────┐
    │ feedback│ ← 本スキル: フィードバック収集
    └────┬────┘
         │ 分析
    ┌────▼────┐
    │ analyze │ ← 本スキル: ギャップ・改善点の特定
    └────┬────┘
         │ 改善案
    ┌────▼────┐
    │ improve │ ← 本スキル: スキル定義・ナレッジの更新
    └────┬────┘
         │ 更新済みスキル
    ┌────▼────┐
    │ 実行    │ → 次回の実行で品質向上
    └─────────┘
```

## 手順

### feedback モード — 実行フィードバック記録

スキル実行直後に呼び出し、結果を記録する。

#### Step 1: フィードバック収集

ユーザーに以下を確認:
1. 実行したスキル名
2. 実行結果の評価（5段階: 1=不十分 〜 5=完璧）
3. 良かった点（そのまま維持すべき内容）
4. 不足していた点（手順に抜けがあった、出力が不十分等）
5. ナレッジベースの過不足（参照情報が足りなかった、古かった等）
6. 予期しない問題（環境依存、前提条件の不備等）

#### Step 2: フィードバックを記録

`~/.claude/knowledge/skill_feedback_log.md` に追記:

```markdown
## {date} — /{skill-name} 実行フィードバック

- **評価**: {1-5}/5
- **良かった点**: {記述}
- **不足点**: {記述}
- **ナレッジ過不足**: {記述}
- **問題**: {記述}
- **改善提案**: {自動生成 or ユーザー入力}
```

### analyze モード — 改善分析

指定スキルのフィードバック履歴を分析し、改善点を特定する。

#### Step 1: フィードバック履歴の集約

`~/.claude/knowledge/skill_feedback_log.md` から該当スキルの全フィードバックを抽出。

#### Step 2: パターン分析

```
=== /{skill-name} 改善分析 ===

[フィードバック統計]
  実行回数: {n}回
  平均評価: {avg}/5
  最頻出の不足点: {item}

[改善カテゴリ]
| カテゴリ | 件数 | 具体例 |
|---------|------|--------|
| 手順不足 | {n} | {例: Step 3 で○○の確認が抜けている} |
| ナレッジ不足 | {n} | {例: DC計算の具体的な数式が欲しい} |
| 出力不足 | {n} | {例: テーブル形式でなくリスト形式で見づらい} |
| 前提条件 | {n} | {例: project.json の○○が必要だが記載なし} |
| 環境依存 | {n} | {例: WSLでは○○が動かない} |

[改善提案（優先順）]
1. [手順追加] Step X に ○○ の確認を追加
   → 影響: {手順の不足を補完}
2. [ナレッジ更新] {knowledge_file} に ○○ を追記
   → 影響: {参照情報の充実}
3. [出力改善] 出力フォーマットに ○○ を追加
   → 影響: {可読性・有用性の向上}
```

#### Step 3: 成熟度レベルの判定

```
現在の成熟度: Level {n}
目標: Level {n+1}
達成に必要な改善: {具体的な条件}
```

### improve モード — 改善適用

analyze の結果に基づき、スキル定義とナレッジベースを実際に更新する。

#### Step 1: 改善計画の確認

analyze の改善提案をユーザーに提示し、実施する項目を選択。

#### Step 2: スキル定義の更新

`.claude/skills/{skill-name}/SKILL.md` を Edit ツールで更新:
- 手順の追加・修正
- 出力フォーマットの改善
- 前提条件の明記
- 関連スキル・ナレッジ参照の追加

#### Step 3: ナレッジベースの更新

`.claude/knowledge/{file}.md` に不足情報を追記:
- 新しい知見・パターンの追加
- 誤りの修正
- プロジェクト固有のカスタマイズ

#### Step 4: 成熟度レベルの更新

`~/.claude/knowledge/skill_feedback_log.md` のスキル成熟度セクションを更新。

#### Step 5: 変更のコミット

```
/commit-change で変更をコミット
コミットメッセージ: "refactor: /{skill-name} スキル改善 — {改善サマリ}"
```

### maturity モード — 全スキル成熟度一覧

```
=== スキル成熟度マップ ===

| レベル | 基準 | スキル数 |
|--------|------|---------|
| L1 Draft | 手順定義のみ | {n} |
| L2 Active | 手順 + ナレッジ参照 | {n} |
| L3 Practiced | 実行フィードバック1回以上 | {n} |
| L4 Mature | フィードバック反映済み + 評価4以上 | {n} |
| L5 Optimized | 継続的改善サイクル確立 | {n} |

[スキル別]
| スキル | レベル | 実行回数 | 平均評価 | 最終改善 |
|--------|-------|---------|---------|---------|
| /fmea | L2 | 0 | — | — |
| /execute-phase | L2 | 0 | — | — |
| /env-check | L2 | 0 | — | — |
...

[成熟度サマリ]
  L1: {n} / L2: {n} / L3: {n} / L4: {n} / L5: {n}
  全体平均: L{avg}
  目標: 主要スキルが L3 以上
```

### cycle モード — 成熟サイクル一括実行

最も改善が必要なスキルを自動特定し、feedback → analyze → improve を一括実行:

1. maturity で全スキルの状態を確認
2. 評価が低い or 未実行のスキルを優先リスト化
3. ユーザーに「最近使ったスキルで改善したいものは？」と確認
4. 選択されたスキルに対して analyze → improve を実行

### tech-refresh モード — 技術最新化

/claude-master と連携し、スキルのドメイン技術を最新化する。

#### Step 1: 対象スキルの特定

- `tech-refresh all` → 全スキルの技術鮮度を確認
- `tech-refresh {skill-name}` → 指定スキルのみ

#### Step 2: ナレッジ鮮度チェック

`~/.claude/knowledge/claude_platform_updates.md` のナレッジ鮮度マップを確認:
- FRESH (30日以内): スキップ
- AGING (31-90日): 軽量チェック推奨
- STALE (91日以上): 巡回必須

#### Step 3: /claude-master domain との連携

```
技術最新化の流れ:

  /skill-evolve tech-refresh {skill}
       │
       ├─ Claude 系スキル → /claude-master scan
       │                     (公式ドキュメント巡回)
       │
       └─ ドメイン系スキル → /claude-master domain {skill}
                              (ドメイン固有の技術巡回)
       │
       ▼
  ナレッジベース更新
       │
       ▼
  スキル手順の反映
       │
       ▼
  成熟度レベル再判定
```

#### Step 4: 更新適用

1. ナレッジベースの更新（新しい知見・ツール・規格の反映）
2. スキル手順への反映（新機能・ベストプラクティスの取り込み）
3. フロントマターの更新（新フィールドの活用）
4. 成熟度の再判定（tech-refresh は L5 到達の加点要素）

#### Step 5: 記録

`~/.claude/knowledge/skill_feedback_log.md` に tech-refresh 履歴を記録:

```markdown
## {date} — /{skill-name} 技術最新化

- **巡回ソース**: {公式docs / ドメイン検索}
- **発見事項**: {新機能・変更点}
- **適用内容**: {スキル・ナレッジへの反映内容}
- **ナレッジ更新**: {更新したファイル}
```

## 成熟度レベル定義

| レベル | 名称 | 基準 | 到達条件 |
|--------|------|------|---------|
| L1 | Draft | 基本手順が定義されている | スキル .md ファイルが存在 |
| L2 | Active | ナレッジベース参照あり | `## ナレッジベース参照` セクションが存在 |
| L3 | Practiced | 実行経験あり | feedback が1回以上記録されている |
| L4 | Mature | フィードバック反映済み | improve を1回以上実施 + 平均評価 4.0 以上 |
| L5 | Optimized | 継続改善確立 | improve 2回以上 + 平均評価 4.5 以上 + ナレッジ更新あり |

## 関連スキル
- /claude-master — 技術巡回・最新化（tech-refresh モードが連携）
- /skill-manage — スキル一覧・雛形生成・品質監査（静的管理）
- /health-check — プロジェクト全体の健全性（スキル活用状況セクション）
- /commit-change — 改善内容のコミット
- /session — セッション管理（/session end でフィードバック記録を推奨）

## Error Handling（エラー時の対応）

| 障害 | 検出方法 | 復旧手順 |
|------|---------|---------|
| 入力データ不足 | 必要ファイル/キーの不在 | 不足項目を報告し、前提スキルの実行を促す |
| 処理中断 | 予期しないエラー | エラー内容を報告。変更途中のファイルがあれば `git checkout` で復元 |

## 注意事項
- feedback は実行直後が最も有効（記憶が新鮮なうちに記録）
- improve はユーザー確認後に実行（自動更新しない）
- skill_feedback_log.md は `~/.claude/knowledge/` に配置（共有ナレッジ、全PJ共通）
- 成熟度は自己申告ではなくフィードバックデータに基づき客観判定
- 全スキルを一度に改善しようとせず、使用頻度の高いものから着手

## ナレッジベース参照

本スキル実行時に以下のナレッジベースを参照して品質を向上:
- `~/.claude/knowledge/skill_lifecycle.md` — スキルライフサイクル管理プロセス、成熟モデル、レビュー基準
- `~/.claude/knowledge/claude_code_ops.md` — スキル設計ベストプラクティス、$ARGUMENTS 動作
- `~/.claude/knowledge/claude_platform_updates.md` — 巡回履歴・ナレッジ鮮度（tech-refresh モード用）

※ `~/.claude/knowledge/` = 共有ナレッジ（全PJ共通）、`.claude/knowledge/` = PJ固有ナレッジ
