## 3. 根本原因分析

- **worktree 未 commit 問題**: `claude -p` の WSL エージェントは Edit/Write ツールでファイルを変更するが、git 操作は行わない。オーケストレーターの merge は committed diff のみ比較する設計だったため、未 commit の変更が検出されなかった。設計段階での前提の見落とし。
- **max_turns/timeout 不足**: 初期値は保守的に設定されていたが、エージェントが PROCESS.md + automotive_spice.md + 複数 process_records を全て読む必要があり、読込だけで 15+ turns 消費する。

## 4. 良かった点

- worktree 分離 + merge + rollback パイプラインは設計通り動作
- 4エージェント並行実行で ~16分で全プロセス改善完了（直列なら ~35分）
- cross-checker がプロセス横断の整合性修正を自動実施
- PA値再アセスメントも4並行サブエージェントで効率的に実施
- 品質ゲート（process_records/ 外の変更ブロック）が正しく機能

## 6. 教訓

- WSL エージェントは「ファイル編集はするが git 操作はしない」という前提でオーケストレーターを設計すべき
- ドライランで検証できない問題（commit 有無）は初回本番実行で必ず発見される — 単一エージェント試行の段階的アプローチが有効だった
- max_turns はプロンプト内のリファレンス読込量から逆算して設定すべき（読込 turns + 改善 turns）
