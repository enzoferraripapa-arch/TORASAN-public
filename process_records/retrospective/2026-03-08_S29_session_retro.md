## 2. 発生した問題

| # | 問題 | 影響 |
|---|------|------|
| 1 | クールダウンのロックファイルパス `/tmp/` が Git Bash 呼び出し間で共有されなかった | 初回修正が無効だった |
| 2 | Notification hook がセッション中に一度も発火していなかった | hook 設定変更がセッション中に反映されない可能性 |
| 3 | matcher を idle_prompt に一時変更→ユーザー指摘で全タイプに戻す手戻り | ユーザー要件を最初に正確に確認すべきだった |
| 4 | 通知機構が3重に存在していることに気づくまで時間がかかった | 段階的調査が必要だった |

## 3. 根本原因分析

- **3重通知**: S25 で「二重鳴き」を Stop + Notification の重複として修正したが、Python monitor と watchdog という別系統の通知機構が残存していた。各機構がいつ導入されたか把握されておらず、統合管理されていなかった
- **hook 未発火**: settings.json の hooks 変更がセッション起動時に読み込まれ、セッション中の変更は反映されない可能性がある（次セッションで検証要）
- **ロックファイル問題**: Git Bash の `/tmp/` はプロセスごとに異なるパスに解決される場合がある。`$HOME` ベースのパスに変更して解決

## 4. 良かった点

- 複数エージェント並行調査で watchdog/monitor/hook の全容を短時間で把握できた
- S25 反省会の過去記録が原因特定の参考になった
- ユーザーの「根本的に間違えてないか？」という指摘が本質的な問題発見につながった

## 6. 教訓

- Windows 環境の通知機構は「何が鳴らしているか」を特定するのが最優先。hook だけでなくデーモン/タスクスケジューラも疑うべき
- Git Bash の `/tmp/` はプロセス間で共有されない場合がある。永続ファイルには `$HOME` ベースのパスを使う
- Claude Code の hooks 設定変更はセッション再起動後に有効になる可能性がある
