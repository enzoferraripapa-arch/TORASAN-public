## 2. 発生した問題

### P-01: Edit前にRead未実施（軽微）
- **事象**: skill-evolve, worktree-cleanup, skill-manage の3ファイルをReadせずにEditしようとしてエラー
- **影響**: 手戻り発生（再読込+再編集）
- **原因**: サブエージェント（Specialist）がファイル内容を報告していたため、メインが「読込済み」と誤認

### P-02: コンテキスト枯渇
- **事象**: 前セッション途中でコンテキストウィンドウを使い切り、セッション継続が必要に
- **原因**: Plan Mode での大規模設計 + サブエージェント結果のメインコンテキスト取り込み

## 3. 根本原因分析

| 原因 | 詳細 |
|------|------|
| ツール制約の認識不足 | Edit ツールは「自分が」Read した履歴を要求。サブエージェントの読込は引き継がれない |
| コンテキスト管理 | 大規模改善はサブエージェント活用でもメインコンテキストを大量消費する |

## 4. 良かった点

| # | 内容 |

| 3 | session-handoff → session の統合で重複解消 + DEPRECATED パスも整備 |

## 6. 教訓

1. **サブエージェントのファイル読込はメインに引き継がれない** — 参照更新のような多ファイル編集はメインで Read→Edit が必須
2. **エージェントチームパターンは設計品質に有効** — Specialist（公式BP） + Reviewer（品質検証）の二面検証が抜け漏れを防ぐ
3. **DEPRECATED スキルは互換性のため一時残存させるのが安全** — 他スキルからの参照を全更新してから削除
4. **検証ゲートを事前定義しておくと品質保証が楽** — Plan 段階で G-01〜G-10 を定義し、実装後に一括検証
